# Architecture

depgraph follows a ports-and-adapters (hexagonal) architecture to keep the analysis core decoupled from any specific language server, file system, or UI delivery mechanism.

---

## C4 Model

### Level 1: System Context

```
[Developer]
    │ depgraph <target-dir>
    ▼
[Code Dependency Graph Explorer]
    ├─── reads ───────────▶ [File System]
    └─── JSON-RPC (LSP) ──▶ [LSP Server]  (gopls / typescript-language-server / etc.)
```

### Level 2: Containers

```
┌──────────────────────────────────────────────┐
│  single binary  (web SPA embedded via go:embed)│
│                                              │
│  CLI (Go)                                    │
│    └─ Analysis Engine (Go)                   │
│         ├─ LSP Client ──▶ [LSP Server]       │
│         └─ HTTP Server                       │
│              ├─ GET /graph                   │
│              ├─ GET /file                    │
│              └─ static (embedded SPA)        │
└──────────────────────────────────────────────┘
    │ opens browser
    ▼
[Browser] ──▶ Web Frontend (React SPA, embedded)
```

**Build flow:** `web/` → `npm run build` → `dist/` embedded into `core/` via `go:embed` → single binary

| Container | Tech | Responsibility |
|---|---|---|
| CLI | Go binary | Argument parsing, engine wiring, browser launch |
| Analysis Engine | Go | Port/adapter orchestration, graph construction |
| LSP Client | Go, JSON-RPC | LSP protocol communication and response conversion |
| HTTP Server | Go, net/http | REST API + embedded SPA delivery |
| Web Frontend | React SPA (embedded) | Graph visualization and code viewer |

### Level 3: Components

**core/**

Organized around three ports, each with a corresponding adapter:

| Port | Adapter | Responsibility |
|---|---|---|
| `AnalyzerPort` | LSP Adapter | Starts LSP server, queries `documentSymbol` + `definition` + `references`, validates responses, builds `Graph` |
| `EditorPort` | FS Adapter | Reads file content from local disk; enforces path-traversal protection |
| `ServerPort` | HTTP Adapter | Registers routes generated by oapi-codegen, serves embedded SPA |

**web/**

| Component | Library | Responsibility |
|---|---|---|
| `GraphCanvas` | React Flow | Node/edge rendering, zoom/pan, click events |
| `CodeViewerPanel` | Monaco Editor | File content display with range highlighting (read-only) |
| API client | fetch | `GET /graph`, `GET /file` |

### Level 4: Key Interfaces

```go
type AnalyzerPort interface {
    Analyze(ctx context.Context, root string) (domain.Graph, error)
}

type EditorPort interface {
    GetFileContent(path string) (string, error)
}

type ServerPort interface {
    Serve(ctx context.Context, graph domain.Graph) error
}
```

```go
type Graph struct {
    Nodes []Node
    Edges []Edge
}

type Node struct {
    ID, Label string
    Kind      NodeKind   // "file" | "symbol"
    Path      string
    Range     *Range
}

type Edge struct {
    ID, From, To string
    Kind         EdgeKind   // "defines" | "references"
    Confidence   Confidence // "exact" | "probable"
}
```

## API

Type definitions are generated from `api/openapi.yaml` for both Go (`core/gen/api.gen.go`) and TypeScript (`web/src/gen/api.ts`). Edit the spec first, then run `make gen`.

| Endpoint | Description |
|---|---|
| `GET /graph` | Returns the full dependency graph as JSON |
| `GET /file?path=...` | Returns the content of a source file |

## Runtime Validation

| Layer | Mechanism | What is checked |
|---|---|---|
| HTTP request | oapi-codegen generated wrapper | Required params, type coercion |
| HTTP response | Go struct + `encoding/json` | Type-safe serialization (compile-time) |
| LSP response | Type assertion + error wrapping | Malformed LSP responses detected early |
| FS access | `filepath.Rel` check | Path traversal prevention |
